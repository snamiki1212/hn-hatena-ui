# ADR: Storybookを導入しない

**日付**: 2026-03-29
**ステータス**: Accepted

## 問題

UIコンポーネントの開発・確認にStorybookを導入すべきか。

## 決定

**Storybookは導入しない。**

## 理由

### プロジェクト特性との不一致

- **Astroコンポーネント (`.astro`) が非対応**: コンポーネントの大半は静的な `.astro` ファイル。StorybookはReact Islands (`.tsx`) しか扱えないため、カバレッジが限定的
- **Pencil (.pen) がカタログの役割を担う**: コンポーネント単位のビジュアル仕様は `.pen` ファイルとしてコロケーション済み
- **コンポーネント数が少ない**: HaHeader, HaEntryCard, HaCommentSection 等 10個程度。Storybookの管理コストに見合わない
- **devサーバーで実データ確認が容易**: `npm run dev` で HN API の実データを使ったUIが即座に確認できる。Storybookのモックデータ管理が不要

### コスト

- Storybook本体 + Astro/React連携設定の導入・維持コスト
- Story ファイル (`.stories.tsx`) の作成・同期コスト
- ビルド時間の増加

## 将来の再検討条件

以下のいずれかに該当した場合、Storybook導入を再検討する:

- React Islands が大幅に増え (20+)、ページ全体を起動せず個別確認したくなった場合
- インタラクティブ要素の状態パターン (展開/折りたたみ、色分け段階等) のカタログ化が必要になった場合
- チームメンバーが増え、コンポーネント一覧の共有手段が必要になった場合
