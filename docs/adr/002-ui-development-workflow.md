# ADR-002: UIコンポーネント開発ワークフロー

## Status

**Accepted**

## Context

はてなブックマーク風UIを再現するにあたり、デザインとコードの一貫性を保ちつつ効率的に開発する手順を定義する必要がある。

デザインの参照元 (はてブ) → 設計ドキュメント → Pencilデザイン → コード実装 という段階を経ることで、各ステップの意図と成果物が明確になる。

## Decision

UIコンポーネントの開発は以下の4ステップで進める。

### ワークフロー全体像

```
Step 1          Step 2           Step 3          Step 4
参照             設計              デザイン          実装
─────────────────────────────────────────────────────────
はてブUIを      DESIGN.md に      Pencil (.pen)    .astro / .tsx
観察・分析       仕様を記述        でビジュアル化     にコード化

成果物:          成果物:           成果物:           成果物:
スクリーンショット  DESIGN.md        .pen ファイル     .astro / .tsx
                                                  コンポーネント
```

### Step 1: 参照 — はてブUIの観察

**目的**: 再現対象のUIを正確に把握する

**作業内容**:
- はてなブックマーク (https://b.hatena.ne.jp/) をブラウザで開く
- 対象コンポーネントのスクリーンショットを取得
- DevTools でカラー・フォント・スペーシング等を確認

**成果物**:
- スクリーンショット → `docs/screenshots/{component-name}.png`

**完了条件**:
- 対象コンポーネントのスクリーンショットが保存されている
- 主要なスタイル値 (色、サイズ、余白) を把握している

---

### Step 2: 設計 — DESIGN.md への仕様記述

**目的**: ビジュアルから読み取った情報をテキストとして構造化し、意図を残す

**作業内容**:
- DESIGN.md に対象コンポーネントのセクションを追加/更新
- 構造 (HTML要素の階層)、デザイン (色・サイズ等)、HNデータとのマッピングを記述
- ASCII図でレイアウトを表現

**成果物**:
- `DESIGN.md` の該当セクション更新

**完了条件**:
- コンポーネントの構造・スタイル仕様がテキストで記述されている
- HN API のどのフィールドがどのUI要素に対応するか明記されている
- DESIGN.md だけを読んでコンポーネントの意図とビジュアルが理解できる

---

### Step 3: デザイン — Pencil (.pen) でビジュアル化

**目的**: テキスト仕様をビジュアルに変換し、実装前にレイアウトを確認する

**作業内容**:
- Pencil でコンポーネントのワイヤーフレームを作成
- DESIGN.md の仕様に基づいてカラー・スペーシングを反映
- コンポーネントディレクトリにコロケーション配置

**成果物**:
- `src/components/ha/{ComponentName}/{ComponentName}.pen`

**完了条件**:
- `.pen` ファイルが対応するコンポーネントディレクトリに存在する
- DESIGN.md の仕様と .pen のビジュアルが一致している

---

### Step 4: 実装 — Astro / React コンポーネント化

**目的**: デザインをコードに変換する

**作業内容**:
- 静的コンポーネント → `.astro` ファイルで実装
- インタラクティブコンポーネント → `.tsx` (React Island) で実装
- DESIGN.md のカラー・スペーシング値をそのまま使用
- HN API データの型定義・接続

**成果物**:
- `src/components/ha/{ComponentName}/{ComponentName}.astro` or `.tsx`

**完了条件**:
- コンポーネントが表示され、DESIGN.md / .pen と見た目が一致する
- HN API のデータが正しくマッピングされている

---

### ステップ間の関係

```
Step 1 (参照)
  │  「何が見えるか」を記録
  ▼
Step 2 (設計)
  │  「何を作るか」をテキストで定義 ← Single Source of Truth
  ▼
Step 3 (デザイン)
  │  「どう見えるか」をビジュアルで確認
  ▼
Step 4 (実装)
     「どう動くか」をコードで実現
```

DESIGN.md が Single Source of Truth として機能する。Pencil やコードで迷ったら DESIGN.md に立ち返る。

### コンポーネント単位での進行

ワークフローはコンポーネント単位で回す。全コンポーネントをStep 1で揃えてからStep 2に進む必要はない。

```
HaHeader:    Step1 → Step2 → Step3 → Step4 ✅
HaEntryCard: Step1 → Step2 → Step3 → Step4 ✅
HaComment:          Step1 → Step2 → ...
```

## Consequences

- 各ステップの成果物が明確なので、進捗が追いやすい
- DESIGN.md をテキストで書くため、PRレビュー時にデザイン意図を議論できる
- Pencil (.pen) もテキストベースなのでGit差分が見える
- ステップを飛ばさない規律が必要 (特にStep 2を省略してPencilに飛びたくなるが、テキスト仕様は後で参照されるので重要)
