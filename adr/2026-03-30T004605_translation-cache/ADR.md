# ADR-004: 翻訳結果のキャッシュ

## Context

現在、ビルドのたびに全ストーリータイトル（最大30件）をMyMemory APIで翻訳している。これにより以下の問題が生じている。

| 問題 | 詳細 |
|------|------|
| API制限超過リスク | 30件 × 平均10 words × 4 builds/day = 1,200 words/day > MyMemory無料枠1,000 words/day |
| 無駄なAPI呼び出し | HNストーリーのタイトルは一度公開されると変わらない。同じ英語タイトルを毎ビルドで再翻訳している |
| ビルド時間の増加 | 30件の並列API呼び出し（100-500ms/件）がビルドごとに発生 |

### 要件

| # | 要件 | 重要度 |
|---|------|--------|
| R1 | 翻訳済みテキストをビルド間で永続化する | 必須 |
| R2 | キャッシュヒット時はAPI呼び出しをスキップする | 必須 |
| R3 | GitHub Actions 環境で動作する | 必須 |
| R4 | キャッシュキーは英語タイトル（翻訳対象テキスト）とする | 必須 |

## Candidates

### 1. ファイルベースキャッシュ（JSONファイルをリポジトリにコミット）

- **仕組み**: `src/data/translation-cache.json` に `{ "en_text": "ja_text" }` 形式で保存。ビルド時に読み込み、キャッシュヒットならAPI呼び出しをスキップ。新規翻訳後にファイルを更新しコミット
- **永続化**: Gitリポジトリ自体がキャッシュストア。GitHub Actions のコミット後に `git push` で保存
- **利点**: シンプル。ビルドツールや外部サービス依存なし。キャッシュ内容をコードレビューで確認できる
- **欠点**: キャッシュ更新のたびに自動コミットが必要。リポジトリ履歴にキャッシュ更新コミットが混入する

### 2. GitHub Actions Cache (`actions/cache`)

- **仕組み**: `actions/cache` アクションでキャッシュファイルを保存・復元。ブランチ単位でキャッシュが管理される
- **永続化**: GitHub インフラ側で管理（7日間未使用で自動削除）
- **利点**: リポジトリ履歴を汚さない
- **欠点**: キャッシュの有効期限がある（7日）。workflow設定の変更が必要。ローカルビルドでは機能しない

### 3. ビルド内インメモリキャッシュ

- **仕組み**: ビルド実行中のNode.jsプロセス内でMapを使って重複翻訳をスキップ
- **永続化**: なし（ビルドプロセスと共に消滅）
- **利点**: 実装が最もシンプル
- **欠点**: ビルド間で永続化されないため根本的な解決にならない。同一ビルド内に同じタイトルが出現しない限り効果なし

## Comparison Matrix

| 観点 | ファイルベース (JSON commit) | GitHub Actions Cache | インメモリ |
|------|------------------------------|---------------------|-----------|
| R1 ビルド間永続化 | ◎ | ◎ | × |
| R2 APIスキップ | ◎ | ◎ | △ |
| R3 GitHub Actions対応 | ◎ | ◎ | ◎ |
| R4 英語テキストをキー | ◎ | ◎ | ◎ |
| ローカルビルド対応 | ◎ | × | ◎ |
| 実装コスト | 低 | 中 | 最低 |
| 期限切れリスク | なし | あり（7日） | - |

## Decision

**ファイルベースキャッシュ（JSONファイルをリポジトリにコミット）を採用する。**

### 理由

1. **完全永続化**: Gitリポジトリに保存するため、期限切れがない。一度翻訳したテキストは永続的にキャッシュされる
2. **ローカル・CI両対応**: ローカルビルドでも同じキャッシュが使える
3. **透明性**: キャッシュ内容をGitで追跡・レビューできる
4. **既存アーキテクチャとの整合性**: SSGのコンセプト（ビルド時処理）と相性が良い

### 実装方針

```
src/data/translation-cache.json   ← キャッシュファイル（リポジトリ管理）
src/api/translate.ts              ← キャッシュ読み書きロジックを追加
```

キャッシュの流れ:

```
ビルド開始
  └─ translation-cache.json 読み込み
      └─ 各ストーリーの翻訳時
          ├─ キャッシュヒット → キャッシュ値を返す（API呼び出しなし）
          └─ キャッシュミス  → MyMemory API呼び出し → 結果をキャッシュに追記
  └─ translation-cache.json 書き込み（新規翻訳分のみ追記）
```

GitHub Actions でのキャッシュ更新:

```yaml
# ビルド後にキャッシュファイルをコミット・プッシュ
- name: Commit translation cache
  run: |
    git config user.name "github-actions[bot]"
    git config user.email "github-actions[bot]@users.noreply.github.com"
    git add src/data/translation-cache.json
    git diff --cached --quiet || git commit -m "chore: update translation cache"
    git push
```

### キャッシュファイルのフォーマット

```json
{
  "Show HN: My new project": "Show HN: 私の新しいプロジェクト",
  "Ask HN: How to learn Rust?": "Ask HN: Rustの学習方法は？"
}
```

## Consequences

- **ポジティブ**: API呼び出し数が大幅削減。レート制限超過リスクがほぼゼロになる。ビルド時間が短縮される
- **ネガティブ**: GitHub Actionsのcronビルド後に自動コミット・プッシュが発生し、リポジトリの履歴にキャッシュ更新コミットが混入する
- **キャッシュサイズ**: HNの上位ストーリーは日々更新されるが、人気ストーリーは再出現することもある。長期運用でJSONファイルが肥大化する可能性があるが、テキストデータのため数年単位では問題にならない見込み
